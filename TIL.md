- アルゴリズムとは手順や計算方法
- 例えると、調理の進め方で、その作り方に効率の良い方法と悪い方法がある
- 探索アルゴリズム、ソートアルゴリズム、暗号化アルゴリズムなどがある
- システムやプロセスの設計に必要
- 重要な理由
    - 処理スピード
    - 作業の標準化
    - 問題発生時の原因特定

- フィズバズ問題は、プログラミング力を測るための問題の一つ

- O記法は、計算時間とデータ量の関係について表した記法
    - O(1): データ量（Input size）が増えても計算量(time)は一定
    - O(n): 計算量はデータ量に比例
    - O(n^2): データ量に対して、計算量は指数関数的に増える
    - O(log n): データ量に対して、計算量が常に半分になっていく
- 問題の規模が大きくなっても対応できる良いアルゴリズムを（最悪O(n)を超えないように）意識

## September 22

- データベース

- データベース：データの保存や検索による取得をする、データ管理システム
- 保管方法：データベースの中に二次元表のテーブルを用いる
- テーブル：主キーが必須、かつ、そのデータが1行

- データベースの設計
- データベースの設計は、開発の流れのうちのシステム設計（設計図の1つにデータベースの設計）で行う
    1. まず、何をデータとして保存するか決める
    2. どのような形式で保存するか決める
    3. テーブルで最後保存

1. まず、何をデータとして保存するか決める
    - 保存するもの
        - システムが使用するもの（商品情報など）
        - ユーザーが登録・操作するもの（氏名など）
    1. これに則って、保存するデータを抽出
    2. テーブルの形にする（テーブル名と、列のデータを抽象化した見出し）＝エンティティ（実物や概念といった実体）を定義

2. どのような形式で保存するか決める
    0. 全部のデータを突っ込み、そのデータに対して、テーブルを突っ込むやり方で作る
        - ここでの問題（小野さんを検索しようとすると）
        - もし小野っていう別の人がいて学部や科目も同じなら一意に特定できない

    1. **主キー**で一意にデータを特定できるように
        - ここでの問題（科目名検索しようとすると）
        - 生徒の科目を知りたいとき、1行のデータの中に2行ある状態（1つのセルに値が2つある状態）で、この値を検索できない（できたとしてもパフォーマンス悪い）

    2. 生徒と科目の組み合わせごとに行を追加
        - つまり、**主キー**があって、主キーが各列の値を一意に特定できるようにする
        - 1つのセルの中に1つの値しか入っていない状態（**第一正規形**）
        - 主キーが決まると他のカラムの値も全部決まる**関数従属性**を満たすようにする

        - ここでの問題（オペレーションミスがあると）
        - 名前を変えたけど、システムの方では不具合があって、列2行あったのに、この下の列しか名前が変わらなかった場合、名前に矛盾が生じてしまう

        - 原因
        - {生徒ID, 科目ID}が主キーで、そのうちの生徒IDが決まると名前が決まる（ {生徒ID}→{名前}の関数従属性がある）状態

    3. テーブルを分割して、全体から、生徒テーブルと履修科目テーブルに分ける
        - 名前を変えたら、**その行の名前だけ**が変わるようにする
        - 名前に矛盾が生じなくなる

        - ここでの問題（オペレーションミスがあると）
        - 履修科目テーブルもまだ問題がある
        - もしユーザーが経済学に変更したら、そのうちの下だけ経済学となったら、科目名に矛盾が生じるかも

        - 原因
        - {生徒ID, 科目ID}が主キーで、そのうちの科目IDが決まると科目名が決まるという関数従属性（{科目ID}→{科目名}）がある状態

    4. テーブルを分割して、履修科目テーブルを履修科目テーブルと科目テーブルに分ける（部分関数従属の場所をテーブル分割）
        - 科目ID2を変更したくなったら、科目名の1つだけを変更すればいい
        - 科目名にも矛盾が生じなくなる
        - 第一正規形から部分関数従属を排除したもの（**第二正規形**）
        - **部分関数従属**：主キーの一部に対して従属する列

        - 問題（学部を登録しようとすると）
        - 生徒の在籍しない学部は登録できない

        - 原因
        - 生徒IDが決まったら、学部IDがわかり、これがわかったら学部名が決まるという、多段階の関数従属があること

    5. テーブルを分割して、学部の部分をテーブルとして切り離す
        - 生徒の在籍しない学部も、学部で独立してテーブルがあるから登録できる
        - 第二正規形から推移的関数従属を排除したもの（**第三正規形**）
        - **推移的関数従属**：テーブル内に段階的な従属関係がある
        - これがあると、後ろの部分を登録できない問題や、データの矛盾の問題も生じうる

    6. 基本OKな状態になる

    - SQLで条件分岐（◯◯だったら××の処理を行う）
    - エリアごとの人口を出したい
        1. ラベルを置き換える
        2. エリアごとに集計

            CASE WHEN pref_name IN('','') THEN 'エリアA'
            WHEN pref_name IN ('', '') THEN 'エリアB'
            ELSE NULL
            END AS カラム名,
            SUM(カラム名)
        FROM
        populations;
        // それぞれのデータに値をつけてSELECTし、新しいカラム名を最後に指定している
        GROUP BY
            CASE WHEN pref_name IN('','') THEN 'エリアA'

    - クエリの中のクエリ
    - クエリ：SELECT文のこと
