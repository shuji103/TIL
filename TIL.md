- クラスとインスタンス
    - クラスは構造の部分で、インスタンスはこの構造をもとに固有の値を入れて出力していったもの
    - クラスをまず作って、そこからAさんの投稿、Bさんの投稿というインスタンスを作っていく
    - class クラス名 {
        // 初期化
        constructor(obj){
            // 要素の取得 (obj.◯◯)
            // 実行部分
            this.関数名();
        }
        // 関数の定義
        関数名(){}
      }
      // インスタンスの生成
      const インスタンス名 = new クラス名({オブジェクトのkey: '値'});

- AccordionUIの処理手順（クラスとインスタンスなしver.）

    js
    1. DOM要素の取得(#js-accordion, aタグ)

    3. 関数の定義（クリックしたら、イベントを殺し（aタグのページ遷移の無効化し）、currentTargetとnextElementSiblingで、クリックされた要素の次の要素(コンテンツ部分)を取得し、そのスタイルが表示状態なら非表示に、非表示状態なら表示させる）

    1. クラス内のconstructorを実行
        1. DOM要素の取得（引数にobjを渡し、インスタンスで指定した部分(hookName, tagName)を呼び、obj.hookName, obj.tagNameで指定した要素(id名, タグ名)を取得）

    - プログラミングの登場について、コンピュータへの命令（理解させること）はフォーマットで対応できたが、コンピュータによる出力（出力内容をこちらが理解すること）は困難だった。そこで、天才がわかるアセンブリ言語、一般人にもわかるコンパイラ言語で対応した

    - インタプリタ言語（JavaScriptなど）は、コンパイルなし
    - API連携で他のアプリケーションの機能を自身のものに取り込める
    - アプリケーション開発の流れは、要件定義（ドキュメント化）- 仕様書 - 実装（デザイン作成・プログラミング） - テスト - リリース - 保守・運用

## September 20
- アルゴリズム
- Paiza

- アルゴリズムとは手順や計算方法
- 例えると、調理の進め方で、その作り方に効率の良い方法と悪い方法がある
- 探索アルゴリズム、ソートアルゴリズム、暗号化アルゴリズムなどがある
- システムやプロセスの設計に必要
- 重要な理由
    - 処理スピード
    - 作業の標準化
    - 問題発生時の原因特定
- 学ぶことで、仕事や日常の中の非合理的な作業を、なんとかできないか常に考え、改善し、時間を節約できる

- フィズバズ問題は、プログラミング力を測るための問題の一つ

- O記法は、計算時間とデータ量の関係について表した記法
    - O(1): データ量（Input size）が増えても計算量(time)は一定
    - O(n): 計算量はデータ量に比例
    - O(n^2): データ量に対して、計算量は指数関数的に増える
    - O(log n): データ量に対して、計算量が常に半分になっていく
- 問題の規模が大きくなっても対応できる良いアルゴリズムを（最悪O(n)を超えないように）意識

- Paizaでのプログラム全体の流れ
    1. 入力される値を保存
    2. 入力された値を処理
    3. 保存した答えを出力
    - まず、コメントでプログラムの流れを書いてから、コードを書いていく
- コードのテストには、paiza.ioを用いて、書いたコードをコピペし、入力値を使って実行
    - 以下の３パターンでテスト
    1. 実際に用意されている入出力例パターン
    2. 入力値の境界値を含むパターン
    3. ランダムパターン
